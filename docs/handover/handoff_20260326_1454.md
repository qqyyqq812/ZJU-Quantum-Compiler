## 【已解决的核心问题】

### 1. V8 训练策略坍缩修复
- **问题**: SWAP 惩罚 -2.0 导致模型完全拒绝探索，策略死锁
- **修复**: 回退惩罚至 -1.0, 门奖励 1.0, 完成奖励 20.0
- **结果**: V8 训练顺利收敛，Early Stop @ep17000, avg_swap=13.0

### 2. 零拷贝推断引擎
- **问题**: Qiskit `QuantumCircuit` 深拷贝 ~10ms/次，Beam Search 中产生 30000+ 次导致无法实用
- **方案**: 开发 `LightweightEnv` (纯 Numpy 数组状态机), clone() 操作 O(1)
- **结论**: 实验证明即便环境零拷贝，树搜索中的 PyTorch 推理次数仍是瓶颈。最终确立 **MTx100 (多起点贪心采样)** 为终极推断核心——每次推理仅需一次 forward pass，无需 clone

### 3. V9 拓扑无关观测编码
- **问题**: obs 使用 N×N 映射矩阵编码 (5Q=71 维, 20Q=841 维), 网络无法跨拓扑迁移
- **方案**: 改为「边驱动」固定维度编码：
  - 每条 SWAP 候选边 4 维特征 (best/worst/mean delta, n_improved)
  - 前沿/lookahead 距离各 20 维
  - 聚合统计量 10 维 + 进度 1 维
  - **总计 291 维，与 N 无关**
- **验证**: 5Q obs_dim=291, 20Q obs_dim=291 ✅

### 4. IBM Tokyo 20Q 拓扑注入
- 在 `topologies.py` 中新增 `ibm_tokyo` (43 条双向边, 直径 4, 平均度 4.3)
- 课程学习 5 阶段全线贯通 (3Q→5Q→5Q→10Q→20Q)

### 5. 后端 API 换核
- `src/server/app.py` 的 `/compile` 和 `/benchmarks` 端点从逐步推演换为 `compile_multi_trial(n_trials=100)`

## 【关键技术决策记录】

| 决策 | 理由 |
|------|------|
| 抛弃 Beam Search / MCTS，选择 MTx100 | 树搜索中 PyTorch forward 调用次数爆炸 (30000+次), 不如 100 次独立贪心采样 |
| obs 从 N²映射矩阵改为边驱动 291 维 | 使同一套网络权重可跨 5Q/9Q/14Q/20Q 拓扑迁移，课程学习真正可行 |
| V8 SWAP 惩罚从 -2.0 退回 -1.0 | -2.0 导致策略完全丧失探索能力，模型选择持续 PASS 死锁 |
| 选择 IBM Tokyo 而非 Grid 5x4 | Tokyo 是真实芯片拓扑，有缺失边和非均匀度，论文价值更高 |
