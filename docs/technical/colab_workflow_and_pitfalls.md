# 量子电路：Colab 工作流与防坑指南

本专题文件旨在为后续 Agent 提供【Colab 自动化云端部署】的标准化流转规范，并罗列我们在项目中踩过的致命底层“坑位”。请继任者在操控环境时引以为戒。

## 1. 原则与标准云端工作流
**绝对红线：禁止让用户在 Colab 直接更改代码逻辑。**

1. **本地强控**：所有的参数修改、逻辑调整，必须先在本地 `projects/量子电路` 修改 `.py` / `.sh` 源文件，提交 Git。
2. **云端无脑拉取**：在 Colab 中，永远依靠 `git clone` 或 `git reset --hard` 在启动前强行覆盖，保证云端与本地分支 100% 同步。
3. **一键执行方案**：Colab 具有断开重建后擦除 `/content` 目录的“失忆特性”。必须让用户每次都能无脑点击 `Run All`，将“克隆代码 -> 装环境包 -> 拉取网盘权重 -> 执行训练”一气呵成。

## 2. 踩坑记录库 (Pitfalls)

### 2.1 Colab 断连与持久化危机
- **坑位**：长时间训练时，Colab 前端资源会被强制回收。重启后不仅代码丢失，几十个小时的 `/models` 权重也会蒸发报错。
- **解法**：必须严格约束 `drive.mount('/content/drive')` 挂载 Google 云盘，且只能使用 **软链接 (symlink `ln -s`)** 的手法将生成的 checkpoint 目录映射到云盘。此外，提供 JS 前台代码防睡眠。

### 2.2 Jupyter (.ipynb) 底层结构崩塌陷阱
- **坑位**：Agent 在底层尝试用内置替换工具 API 强行去修改 `.ipynb` 文件里的代码格子。
- **解法**：`.ipynb` 内部包裹的是深度嵌套配置与转义符 JSON。**严禁用底层 API 替换/修改 `.ipynb` 源文件节点！** 最安全的做法是剥离执行代码：将核心参数全扔进 `.py` 或纯净的 `bash script`，让 Notebook 退化成只有两行长周期的只读启动器（比如专门撕裂出的 `A_baseline.ipynb` 和 `B_fallback.ipynb`）。

### 2.3 RL 早停机制重灾区 (Early Stopping)
- **坑位**：按照常规分类任务的耐心参数（比如 Patience设为10），复杂的 20Q 强化模型在早期探索的盲目震荡期，会在刚刚跑到 2 万局时被无情判定为退化并强行中断 `Done`。
- **解法**：针对极其离散、高震荡的 RL 路由任务，早停的 Patience 必须被极度放宽到 100 步以上甚至更多。
